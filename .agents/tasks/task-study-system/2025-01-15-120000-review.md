# Personal Study System — full greenfield implementation

A complete single-page study management app for a Bangladeshi college student, built as a React + TypeScript SPA with Vite, Tailwind CSS, shadcn/ui components, and Zustand stores persisted to localStorage. The system covers daily logging, task management (list and Kanban board), three study trackers (college/tuition/self-study), spaced-repetition revision, exams with countdown, a weekly routine, calendar, stats/streaks, and a global command-palette search. A Supabase-ready SQL schema is included for future backend migration.

**Watch for:** Seed data generates fresh UUIDs on every cold load, so column/subject IDs in localStorage will never match IDs generated in a subsequent session if the store is cleared and re-seeded (confirmed). The revision store's `getCompletedToday` relies on back-calculating the review date from `nextReviewDate - interval`, which silently breaks when a user reschedules or skips an item (confirmed). The undo-toast DOM manipulation bypasses React's lifecycle entirely and leaks global style/event state (confirmed).

## High-level view

The data model is a flat set of TypeScript interfaces with soft-delete (`deletedAt`) used consistently across all entities except routine entries (which hard-delete). Each entity family gets its own Zustand store with `persist` middleware writing to a dedicated localStorage key. This provides offline-first simplicity but means cross-store consistency (e.g. deleting a subject doesn't cascade to tasks referencing it) is never enforced.

State management follows a "selector per accessor" pattern: each store exposes both raw arrays and getter methods (`getActiveTasks`, `getEntriesByType`). The getters call `get()` inside the store rather than using selectors, which means they re-run on every render rather than benefiting from Zustand's reference-equality optimization.

The task system offers two views — a time-bucketed list view and a @dnd-kit Kanban board — both operating over the same filtered task set. Drag-and-drop moves tasks between columns; sorting within a column is not persisted (tasks appear in array order). The board columns are themselves user-configurable via Settings.

The spaced-repetition system uses a fixed `[1, 3, 7, 14, 30]` day interval array and advances the pointer on each review. Completing a college tracker topic automatically enqueues a revision item. The "completed today" calculation reverse-engineers when a review happened by subtracting the interval from `nextReviewDate`, which is fragile when any manual reschedule has occurred.

The imperative undo-toast creates and removes raw DOM nodes, injects a global `<style>` tag, and holds a module-level singleton reference. This sidesteps React's rendering model and can leave orphaned DOM nodes if the component unmounts during the 5-second window.

The SQL schema maps 1:1 to the frontend data model, adds RLS policies per table, and uses `auth.uid()` for tenant isolation. Tags use a junction table (`task_tags`) rather than the frontend's array-of-IDs approach, which will require a migration adapter when the backend is wired up.

## Seed data identity instability

`src/data/seed.ts` calls `generateId()` (which is `uuid.v4()`) at module evaluation time to build default subjects, columns, and tags. Zustand's `persist` middleware only uses these defaults when no localStorage entry exists for the store. If a user clears their browser data, or the app is loaded in a new browser, all seeds get fresh IDs. Any tasks that were exported, shared, or referenced by ID are now orphaned.

```ts
// seed.ts — every import generates new IDs
export const defaultSubjects: Subject[] = [
  createSubjectWithPapers("Bangla", ["1st Paper", "2nd Paper"]),
  // ...
];
```

Use fixed UUIDs for seed data (e.g. `"00000000-0000-0000-0000-000000000001"`).

## Revision schedule's fragile "completed today" heuristic

`useRevisionStore.getCompletedToday` determines whether an item was reviewed today by computing `nextReviewDate - intervals[reviewCount]`:

```ts
const interval = s.intervals[Math.min(s.reviewCount, s.intervals.length - 1)];
const reviewedAt = addDays(new Date(s.nextReviewDate), -interval);
return isWithinInterval(reviewedAt, { start: todayStart, end: todayEnd });
```

This breaks in two scenarios: (1) `skipItem` sets `nextReviewDate` to tomorrow without incrementing `reviewCount`, so the back-calculation produces a stale date; (2) `reschedule` sets an arbitrary date. Without an explicit `lastReviewedAt` field, the "Completed Today" section shows phantom entries or misses real ones.

## Imperative undo toast outside React's tree

`useToastUndo.ts` manually creates DOM elements via `document.createElement`, appends them to `document.body`, and tracks them in a module-scoped `let toastElement`. The `<style>` element with `id="undo-toast-styles"` is injected once and never removed. If the React app unmounts (e.g. during HMR in dev), the toast DOM lingers. Because the `onUndo` closure captures the store's `restoreFn` at call time, any store state change during the 5-second window means undo restores into a potentially stale slice. The project already depends on `@radix-ui/react-toast` — the Radix Toast primitive would eliminate these lifecycle issues.

## Store getter re-render cost

Every Zustand store exposes methods like:

```ts
getActiveTasks: () => get().tasks.filter((t) => t.deletedAt === null),
```

Components call these as `useTaskStore((s) => s.getActiveTasks())`. The method reference is stable so Zustand's subscription fires on every state change, but the return value is a new array each time. The result: all components subscribing to a getter re-render on any mutation to that store's state, regardless of which slice changed. For a personal app with tens of items this won't cause visible lag, but it compounds as data grows toward the Supabase migration.

## Deletion semantics asymmetry

All stores except `useRoutineStore` use soft-delete:

```ts
// useRoutineStore — hard delete
deleteEntry: (id) =>
  set((state) => ({ entries: state.entries.filter((e) => e.id !== id) })),
```

Routine entries cannot be undone, and a future "trash/archive" feature would need special-casing for this store alone.

## College tracker auto-enqueues revision without dedup

When a topic cycles to "completed", `CollegeTracker.tsx` calls `addRevision` unconditionally:

```ts
if (next === "completed") {
  addRevision({ topicName, subjectId, paperId, firstStudiedAt: ... });
}
```

Cycling a topic back to "in_progress" and then to "completed" again creates a duplicate revision entry. There's no uniqueness check on `(topicName, subjectId, paperId)` in the revision store.

## Mobile navigation truncation

`AppLayout` renders only the first 5 nav items in the mobile bottom bar (`navItems.slice(0, 5)`). Revision, Exams, Routine, Calendar, and Settings are completely unreachable from mobile. No hamburger menu, drawer, or "More" button exists as an alternative path.
